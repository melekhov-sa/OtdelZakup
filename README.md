# OtdelZakup

Веб-приложение + JSON API для загрузки, просмотра и преобразования Excel-файлов (.xlsx) отдела закупок.

## Запуск локально

```bash
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

Приложение доступно на http://localhost:8000

## Запуск через Docker Compose

```bash
docker compose up --build
```

## Тесты

```bash
pytest -q
```

## Авто-детект табличной части

Парсер автоматически определяет структуру загруженного .xlsx файла:

1. **Поиск заголовков** — сканирует первые 80 строк, для каждой считает "табличный скор" (+1 за код, +1 за наименование, +1 за количество). Выбирает строку с максимальным скором (>= 2).

2. **Fuzzy matching** — сравнивает заголовки колонок с синонимами через rapidfuzz (порог > 70):
   - Код: `код`, `артикул`, `№`, `номер`
   - Наименование: `номенклатура`, `наименование`, `товар`, `позиция`
   - Количество: `заказ`, `кол-во`, `количество`, `потребность`, `к заказу`

3. **Многострочная шапка** — если следующая строка содержит "шт", "ед.изм" и т.п., заголовки склеиваются.

4. **Эвристика по содержимому** — если количество не определено по заголовку, ищет колонку с >= 60% числовых значений. Если наименование не найдено — колонку с самой длинной средней строкой.

5. **Ручной выбор** — если авто-детект не уверен, пользователь видит страницу выбора колонок с предпросмотром таблицы и 3 выпадающими списками.

## Веб-интерфейс

- `GET /` — **главная страница**: форма загрузки (вкладки «Файл» / «Текст») + настройки системы
- `POST /upload` — загрузка .xlsx, авто-детект → таблица или страница выбора колонок
- `POST /text-input` — ввод позиций текстом → разбор → таблица предпросмотра
- `POST /apply-columns` — ручной выбор колонок (если авто-детект не удался)
- `POST /transform` — преобразование с выбранными полями, две таблицы рядом
- `GET /download/{file_id}/{token}` — скачать результат преобразования в .xlsx

### Сценарий демо

1. Открыть http://localhost:8000
2. В нижней зоне «Настройки системы» можно сразу посмотреть и настроить правила готовности и справочник стандартов — без загрузки файла
3. Загрузить .xlsx файл с метизами
4. Если колонки определились автоматически — сразу видите таблицу
5. Если нет — выберите колонки вручную из выпадающих списков, нажмите "Применить"
6. Отметить нужные поля (диаметр, длина, класс прочности, покрытие, ГОСТ и т.д.)
7. Нажать "Преобразить" — увидеть исходник слева, результат справа, строки подсвечены по статусу
8. Посмотреть статистику, отфильтровать строки чекбоксами
9. Нажать "Скачать все строки .xlsx" или "Скачать только строки без проверки .xlsx"

## Мульти-источниковое извлечение (Signal Merging)

Сервис умеет учитывать **отдельные колонки** в загруженном файле, если клиент вынес атрибуты в выделенные столбцы (например: «Стандарт», «Класс прочности», «Примечание»).

### Авто-детект дополнительных колонок

При разборе файла парсер, помимо основных колонок (Код, Наименование, Количество), ищет:

| Колонка | Синонимы для fuzzy-match |
|---|---|
| Стандарт | `стандарт`, `гост`, `iso`, `дин`, `din` |
| Класс прочности | `класс прочности`, `класс`, `прочность` |
| Примечание | `примечание`, `комментарий`, `требование`, `тд`, `условие` |

Найденные колонки сохраняются как `standard_raw`, `strength_raw`, `note_raw` в DataFrame.

### Приоритеты источников

При трансформации данные извлекаются из нескольких источников с приоритетом:

1. **Выделенная колонка** (highest) — если в файле есть отдельный столбец «Стандарт» или «Класс прочности», значение берётся оттуда
2. **Колонка наименования** — regex-извлечение из текста названия товара
3. **Колонка примечания** (lowest) — если в «Примечание» упомянут стандарт или класс прочности

Пример: если в наименовании нет класса прочности, но в колонке «Класс прочности» указано `8.8` — значение будет `8.8`.

### Нормализация стандартов

- `ГОСТ Р ИСО 4014-2013` → `ГОСТ Р ИСО 4014-2013` (сохраняется полностью)
- `ИСО 4032` → `ISO 4032` (если не является частью ГОСТ Р ИСО)
- `11371-78` (голый номер) → `ГОСТ 11371-78`
- `8` или `10` (класс прочности) → `8.8` / `10.9`

## Правила готовности (Readiness Rules)

Система настраиваемых правил, которые определяют, когда строка считается «Не требует проверки». Правила хранятся в SQLite и управляются через веб-интерфейс.

### Как работает

1. Для каждой строки определяется `item_type` (болт, гайка, шайба и т.д.)
2. Ищется первое активное правило с совпадающим `item_type` (по приоритету)
3. Если нет — используется правило по умолчанию (item_type = NULL)
4. Проверяются обязательные поля (`require_fields`) правила:
   - Все поля заполнены → **«Не требует проверки»** (ok, зелёная подсветка)
   - Не хватает size или qty → **«Требуется вручную разобрать»** (manual, красная подсветка)
   - Не хватает прочих полей → **«Требуется просмотреть»** (review, жёлтая подсветка)
5. В колонке «Причина» отображаются недостающие поля

### Управление правилами

- `GET /readiness` — список всех правил с кнопками включения/выключения
- `GET /readiness/new` — форма создания правила
- `POST /readiness/create` — создать правило
- `GET /readiness/{id}/edit` — форма редактирования
- `POST /readiness/{id}/update` — обновить правило
- `POST /readiness/{id}/toggle` — включить/выключить правило

### Правила по умолчанию (seed)

| Имя | Тип изделия | Обязательные поля | Приоритет |
|---|---|---|---|
| По умолчанию | Все типы | Наименование, Количество, Ед. | 0 |
| Шайба | шайба | Размер, Количество | 10 |
| Гайка | гайка | Размер, Количество | 10 |
| Болт | болт | Размер, Количество | 10 |
| Винт | винт | Размер, Количество | 10 |
| Саморез | саморез | Размер, Количество | 10 |

### Доступные поля для правил

size, qty, uom, name, code, item_type, strength, coating, gost, iso, din.

Не хватает `size`, `qty` или `uom` → статус «Требуется вручную разобрать» (manual).
Не хватает других полей → статус «Требуется просмотреть» (review).

## Правила проверки (Validation Rules)

Правила проверки выявляют **противоречия и ошибки** в данных: запрещённые поля, обязательные поля для конкретного типа. Хранятся в той же SQLite БД (таблица `validation_rule`).

В отличие от правил готовности, правила проверки **не переводят строку в статус «Не требует проверки»** — они указывают на проблему и могут принудительно выставить статус «Требуется просмотреть» или «Требуется вручную разобрать».

### Поля правила

| Поле | Описание |
|---|---|
| `item_type` | Тип изделия (NULL = для всех типов) |
| `require_fields` | Поля, которые должны быть заполнены |
| `forbid_fields` | Поля, которые должны быть пустыми |
| `force_status` | Статус при срабатывании: `review` или `manual` |
| `priority` | Порядок проверки (меньше = раньше) |

### Управление правилами проверки

- `GET /rules` — список всех правил
- `GET /rules/new` — форма создания
- `POST /rules/create` — создать правило
- `GET /rules/{id}/edit` — форма редактирования
- `POST /rules/{id}/update` — обновить
- `POST /rules/{id}/toggle` — включить/выключить

## Нормализованное наименование

Дополнительная колонка «Нормализованное наименование», которая формируется по настраиваемому шаблону. Не входит в базовый набор извлекаемых полей — включается отдельной галочкой на экране преобразования.

### Как включить

На экране «Извлечение полей» (после загрузки файла) среди чекбоксов появляется пункт **«Нормализованное наименование»** — он отображается только при наличии активного шаблона. Если галочка отмечена, в таблицу результатов и в скачиваемый .xlsx добавляется соответствующая колонка.

### Шаблон наименования

Шаблон задаёт формат результирующей строки через переменные в фигурных скобках:

```
{item_type} {size} {strength} {standard}
```

Пустые переменные убираются автоматически, лишние пробелы схлопываются.

| Переменная | Описание |
|---|---|
| `{item_type}` | Тип изделия (болт, гайка…) |
| `{size}` | Размер (M12x80) |
| `{strength}` | Класс прочности (8.8) |
| `{standard}` | Стандарт: первое непустое из ГОСТ → ISO → DIN |
| `{coating}` | Покрытие (цинк, нержавейка…) |
| `{qty}` | Количество |
| `{code}` | Код позиции |
| `{name}` | Исходное наименование |

### Где менять шаблон

Управление шаблонами доступно через:
- Вкладку **«Шаблон наименования»** на главной странице (`GET /`)
- Прямой ссылкой `GET /name-templates`

| Роут | Действие |
|---|---|
| `GET /name-templates` | Список шаблонов |
| `GET /name-templates/new` | Форма создания |
| `POST /name-templates/create` | Создать шаблон |
| `GET /name-templates/{id}/edit` | Форма редактирования |
| `POST /name-templates/{id}/update` | Обновить шаблон |
| `POST /name-templates/{id}/activate` | Сделать единственным активным |
| `POST /name-templates/{id}/toggle` | Включить / выключить |

При нажатии «Активировать» все остальные шаблоны деактивируются. По умолчанию устанавливается один шаблон: `{item_type} {size} {strength} {standard}` с приоритетом 1.

## Справочник стандартов (Standard Reference)

База данных стандартов (DIN/ISO/ГОСТ), связывающая код стандарта с типом изделия. Хранится в той же SQLite БД.

### Как улучшает распознавание

После извлечения стандарта из текста строки (например `DIN 934`) система ищет его в справочнике:

1. **Автозаполнение item_type** — если в тексте нет ключевого слова типа изделия (болт/гайка/...), но в строке есть `DIN 934`, а в справочнике `DIN 934 → гайка`, то `item_type` автоматически подставляется. Затем правила готовности могут найти более точное правило.

2. **Проверка несоответствия** — если из текста извлечён `item_type = "болт"`, но стандарт `DIN 934` соответствует "гайка", в колонку «Причина» добавляется предупреждение и статус принудительно становится минимум «Требуется просмотреть».

3. Колонка `item_type_source` отражает источник значения: `"из текста"` или `"из стандарта"`.

### Управление справочником

- `GET /standards` — список стандартов с поиском и фильтром по виду
- `GET /standards/new` — форма добавления нового стандарта
- `POST /standards/create` — сохранить
- `GET /standards/{id}/edit` — редактировать
- `POST /standards/{id}/update` — сохранить изменения
- `POST /standards/{id}/toggle` — включить/выключить стандарт

### Как добавить новый стандарт через UI

1. Открыть http://localhost:8000/standards
2. Нажать **«+ Создать стандарт»**
3. Выбрать вид (GOST/ISO/DIN), ввести код (без префикса, например `7798-70`)
4. Указать тип изделия из выпадающего списка
5. Нажать **«Создать»**

## Количество и единицы измерения

Парсер автоматически распознаёт количество и единицы измерения в нескольких форматах.

### Поддерживаемые форматы (функция `parse_qty_uom`)

| Формат | Пример | qty | uom |
|---|---|---|---|
| Число + пробел + ед. | `1500 шт` | 1500 | шт |
| Число + ед. без пробела | `8шт` | 8 | шт |
| Дробное с запятой | `2,5 кг` | 2.5 | кг |
| В скобках в конце | `Гайка М10 (200 шт)` | 200 | шт |
| В конце наименования | `Болт М12х80 50шт` | 50 | шт |

### Нормализация единиц

| Исходные формы | Нормализованная форма |
|---|---|
| шт, штука, штуки, штук | шт |
| кг, килограмм, килограмма | кг |
| г, гр, грамм, граммов | г |
| м, метр, метра, метров | м |
| мм | мм |
| л, литр, литра | л |
| уп, упак, упаковка | уп |
| компл, комплект | компл |
| пач, пачка | пач |

### Автодетекция источника

**Вариант A (объединённая колонка).** Если заголовок колонки похож на «Количество» / «Заказ» и ≥50% значений содержат текст вида `N ед.` — парсер автоматически разбивает каждое значение на `qty` и `uom`.

**Вариант B (из наименования).** Если колонка qty отсутствует, парсер проверяет конец поля «Наименование»: число + единица в конце строки или в скобках. Если найдено — qty и uom заполняются, из наименования удаляется найденный фрагмент.

При дробном количестве (например, 2.5 кг) значение сохраняется как вещественное число.

## Ввод текстом

На главной странице есть вкладка **«Вставить текст»** — альтернатива загрузке Excel. Список позиций вставляется напрямую, система разбирает его и направляет в тот же pipeline преобразования.

### Поддерживаемые форматы

| Формат | Пример |
|---|---|
| Нумерованный список | `1. Болт М12х80 DIN 931 — 100 шт` |
| Тире/дефис после наименования | `Гайка М12 DIN 934 - 50 шт` |
| Количество на следующей строке | `Болт М16х120`<br>`Количество: 200 шт` |
| Без количества (plain text) | `Болт М12х80` |

### Строгое правило qty + uom

Количество и единица измерения должны быть указаны **оба** — иначе оба остаются `None`. Строка без единицы (`Болт М12 - 100`) не получит статус «ok», а попадёт в «Требуется вручную разобрать» (нет `uom`).

### Сервисные строки (примечания)

Строки, начинающиеся с «все», «можно», «особые», «требование», «примечание», «комментарий», «итого», «p.s.» — собираются в поле `note_raw` и применяются ко всем строкам заявки.

### Тысячные разделители

Русский формат `10 000 шт` поддерживается — пробел как разделитель тысяч распознаётся корректно.

### Пример

```
1. Болт М12х80 DIN 931 — 100 шт
2. Гайка М12 DIN 934 — 200 шт
3. Шайба 12 DIN 125 — 400 шт
все в горячем цинке
```

Результат: 3 позиции, `note_raw = "все в горячем цинке"` на каждой строке.

### Роут

`POST /text-input` (form-data, поле `text`) — возвращает ту же страницу `view_raw.html`, что и загрузка Excel.

| **Б. Что распознано** | Значения всех полей после экстракции: тип изделия, размер, диаметр, длина, класс прочности, покрытие, стандарты |
| **В. Справочники** | Найден ли стандарт в справочнике, какой тип изделия из него выведен, конфликты |
| **Г. Готовность** | Какое правило готовности применилось, обязательные поля, чего не хватает, базовый статус |
| **Д. Правила проверки** | Список сработавших validation-правил с причиной и принудительным статусом |
| **Е. Итог** | Итоговый статус и итоговая строка причин |

### Как открыть

В «Преобразованной таблице» первый столбец каждой строки содержит её номер и ссылку 🔍. Клик открывает модальное окно без перехода на другую страницу. Закрыть — кнопкой ✕, клик вне карточки или Escape.

### API

```
GET /files/{file_id}/rows/{row_number}/analysis
```

Возвращает JSON с ключами `row_number`, `raw_inputs`, `extracted_fields`, `enrichment`, `readiness`, `validation`, `final`. Если для файла ещё не выполнялось преобразование, возвращает `404` с `{"error": "..."}`.

Трейс хранится в `{CACHE_DIR}/{file_id}/traces.json` и перезаписывается при каждом новом вызове `/transform`.

## Как определяется статус строки

Итоговый статус каждой строки формируется **последовательно из трёх источников** — никакой скрытой логики, никакого «доверия».

### Порядок источников

| # | Источник | Условие срабатывания |
|---|---|---|
| 1 | **Пустое наименование** | name пустой, "-" или "—" → сразу **manual**, причина "Пустое наименование". Дальнейшие проверки не выполняются. |
| 2 | **Правила готовности** | Определяют базовый статус по обязательным полям. Если все правила отключены → базовый статус **ok**. Если нет подходящего правила → **manual**. |
| 3 | **Правила проверки** | Могут ухудшить базовый статус (ok→review, review→manual) и добавить причину. Если все правила проверки отключены → этот шаг пропускается. |

Статус может только ухудшаться: **ok → review → manual**. Ни один источник не может повысить статус, выставленный более ранним источником.

### Что происходит при отключении правил

- **Все правила готовности выключены** — каждая строка с непустым наименованием получает статус **ok** (без анализа полей).
- **Все правила проверки выключены** — шаг валидации пропускается полностью, базовый статус не меняется.
- **Пустое наименование** — всегда **manual**, независимо от состояния правил.

### Почему это важно (детерминированность)

Итоговый статус строки определяется **только** активными правилами и содержимым строки. Два файла с одинаковыми данными и одинаковыми правилами дадут одинаковые статусы. Блок «Анализ строки» (🔍) всегда объясняет, почему строка получила тот или иной статус.

### Confidence (вспомогательная метрика)

После преобразования каждая строка дополнительно получает оценку confidence (0–5):
- +1 если найден diameter
- +1 если найден length
- +1 если найден strength
- +1 если найден coating
- +1 если найден gost или din

Confidence **не влияет на статус** — это информационная метрика для ориентации. Статус определяется исключительно правилами (см. выше).

Над таблицей результатов отображается блок статистики и чекбоксы фильтрации по статусу (все подписи на русском).

Два варианта скачивания: все строки или только строки без проверки.

## JSON API v1 (для интеграции с 1С)

### POST /api/v1/upload

Загрузка файла. `multipart/form-data`, поле `file` (.xlsx).

Ответ (авто-детект успешен):
```json
{"file_id": "abc123...", "filename": "data.xlsx", "rows_total": 150, "columns": ["code", "name", "qty", "uom"], "needs_column_selection": false}
```

Ответ (нужен ручной выбор колонок):
```json
{"file_id": "abc123...", "filename": "data.xlsx", "needs_column_selection": true, "preview_rows": [...], "num_columns": 5, "detected": {"name_idx": null, "qty_idx": 2, "code_idx": null, "header_row": 0, "score": 1}}
```

Ошибка: `400 {"error": "..."}`.

### POST /api/v1/apply-columns

Ручной выбор колонок. `Content-Type: application/json`.

Тело запроса:
```json
{"file_id": "...", "name_col": 2, "qty_col": 4, "code_col": 0, "header_row": 3}
```

`code_col` — опционально (-1 = нет). `qty_col` — опционально (-1 = нет).

Ответ:
```json
{"file_id": "...", "filename": "data.xlsx", "rows_total": 150, "columns": ["code", "name", "qty", "uom"]}
```

### GET /api/v1/preview/{file_id}?limit=200

Предпросмотр исходных данных. `limit` — опционально (по умолчанию 200).

Ответ:
```json
{"file_id": "...", "rows_total": 150, "limit": 200, "columns": ["code", "name", "qty", "uom"], "rows": [["001", "Болт M12", 100, "шт"], ...]}
```

`rows` — массивы (не объекты). Если `file_id` не найден: `404 {"error": "not found"}`.

### POST /api/v1/transform

Извлечение полей. `Content-Type: application/json`.

Тело запроса:
```json
{"file_id": "...", "fields": ["diameter", "strength"], "limit": 200}
```

Доступные поля: `diameter`, `length`, `size`, `strength`, `coating`, `gost`, `iso`, `din`, `tail_code`, `screw_diameter`, `thread_type`, `item_type`, `standard_raw`, `strength_raw`, `note_raw`. `limit` — опционально (по умолчанию 200).

Ответ:
```json
{"file_id": "...", "rows_total": 150, "fields": ["diameter", "strength"], "columns": ["code", "name", "qty", "uom", "Диаметр", "Класс прочности", "confidence", "status"], "rows": [["001", "Болт M12x80 8.8 оц ГОСТ 7798-70", 100, "шт", "M12", "8.8", 5, "ok"], ...]}
```

Колонки `confidence` (0–5), `status` ("ok" / "review" / "manual") и `reason` добавляются всегда. Статус определяется правилами готовности.

Если `file_id` не найден: `404 {"error": "not found"}`.
